# v0.1.x detection-rule strategy synthesis

Date: 2026-05-04. Author: Sankalp Gilda. Methodology: 4-persona deep-research session
(`registry-maintainer`, `working-security-engineer`, `idna-internals-skeptic`,
`real-world-callsite-cataloguer`) under `~/.claude/skills/deep-research-skill/`,
plus a local empirical sweep against three already-cloned Go corpora and the
matchlock SDK's internal `idnarecheck` analyzer.

## TL;DR

The v0.1.0 detection-rule shapes (Semgrep taint with two labels, CodeQL stateful
taint with two flow states) are **directionally correct** for the bug class but
the operational story is sharper than v0.1.0 admits. The v0.1.x shipping
recommendation:

1. **CodeQL is the primary recall vehicle.** Its `TaintTracking::GlobalWithState`
   is inter-procedural by default, propagates through one-deep wrappers like
   `idnaASCII` without explicit `isAdditionalFlowStep`, and is the closest to
   the structural-correctness behaviour of the matchlock SDK's internal Go
   analyzer. Verify it fires on `x/net/http/httpproxy.canonicalAddr` before
   submitting to the community pack.
2. **Semgrep OSS taint stays as the high-precision secondary sweep.** It catches
   direct-call sites with explicit untrusted-input shapes. It produces zero
   alerts on real production Go because real production Go wraps. That is
   honest signal at the OSS tier; do not paper over it with a high-recall
   structural rule that pathologises legitimate registrar code.
3. **Do not ship a blanket structural "ToASCII without recheck in same block"
   rule.** The empirical FP density is intolerable: 30 of 31 production
   callsites lack the recheck, but the vast majority are legitimate (PSL
   walkers, registrar pipelines, TLS-SNI normalization where the result is
   never network-routed on attacker input). A blanket structural rule pretends
   the recheck is a universal contract; it is not.
4. **Document the tool stratification clearly in the README** so registry
   reviewers understand the design choice and operators know which tool gives
   them which signal.

The two-tool stratification (CodeQL inter-procedural recall + Semgrep OSS
direct-call precision) is the same shape `BadRedirectCheck.ql` and
`IncompleteUrlSchemeCheck.ql` use in the CodeQL community pack today. It is
a defensible registry submission narrative.

## Why the v0.1.0 0/0/0 corpus result is the right answer

The 660 MB sweep against `golang/go`, `kubernetes/kubernetes`, and
`prometheus/prometheus` produced zero alerts across the OSS, Pro, and
experimental yamls. The natural reaction is "the rule under-fires."

The empirical truth is sharper. There are exactly two production callsites
of `idna.Lookup.ToASCII` (or any UTS-46-mapping profile) anywhere in the 660
MB sweep. Both are wrapped by a small `idnaASCII` helper:

- `golang/go: src/net/http/request.go:799`
- `golang/go: src/vendor/golang.org/x/net/http/httpproxy/proxy.go:312`

Other idna usages in the sweep call package-level `idna.ToASCII`, which
dispatches to the Punycode profile (no UTS-46 NFKC mapping, no smuggle
surface). Those are correctly out of scope.

The OSS taint rule cannot step through the `idnaASCII` wrapper because OSS
Semgrep is intra-procedural. So zero alerts in the OSS sweep is not a rule
defect; it is the OSS tier's honest output for a corpus where the only
in-scope callsites are wrapped. Pro Engine with `interfile: true` would
catch the wrapper. CodeQL with `TaintTracking::GlobalWithState` would catch
it too, by default, with no Pro engine licence required. The right fix is
not to make the OSS rule fire louder by relaxing the source set; the right
fix is to lead with CodeQL in the registry submission.

## Why a structural "ToASCII without recheck" rule is wrong

Persona P4 catalogued 31 production callsites of `idna.*ToASCII` across 19
distinct repos (caddy, vault, certmagic, ooni/probe-cli, smallstep,
sing-box, mattermost, hostmatcher, lorawan-stack, tlsproxy, datadog-agent,
cloudflared, mosn, safebrowsing, sniproxy, q, whatwg-url, plus the Go
stdlib and x/net mirrors). Of those:

| Class | Count | Description |
|---|---|---|
| (a) direct call | 18 | `idna.Lookup.ToASCII(input)` directly in a function whose source is identifiable |
| (b) one-deep wrapper | 6 | small helper named `idnaASCII` / `toASCII` returning the call result raw |
| (c) multi-deep / conditional wrapper | 6 | helper that branches on `isASCII` / `len` and only calls ToASCII on a sub-path |
| (d) post-call IP recheck present | 1 | `google/safebrowsing/urls.go:260` — and even this uses `strings.Trim` (both-side) not `TrimRight`, with an in-house `parseIPAddress` not stdlib `netip.ParseAddr` |

A blanket structural rule "any ToASCII without TrimRight + netip.ParseAddr in
the same block" would fire on 30 of 31 production callsites. The 30 are not
all bugs. Many are PSL / registrar / TLS-cert-manager code that does not
network-route the result on attacker input, so the missing recheck is not a
smuggle vector. The rule would be ~95% false-positive in the strict
"vulnerable to UTS-46 smuggling" sense, and operators would lose trust
within one CI cycle.

P3 reinforced this: weppos/publicsuffix-go, x/net/publicsuffix, cloudflare-go
issue #688, and every PSL-driven cookiejar codepath are documented
non-network-routing IDNA users. A rule that flags them as smuggle bugs
pathologises the documented library contract. This is the same reason no
prior IDNA-class CVE (CVE-2021-29923, CVE-2024-12224, CVE-2024-3651,
CVE-2026-39821) shipped with a Semgrep or CodeQL rule attached: the
detection space is too noisy without inter-procedural taint scoping.

## Why blanket-structural fires twice on the canonical wrapper

The structural rule prototype tested locally fires on
`x/net/http/httpproxy/proxy.go:312` and `src/net/http/request.go:799` because
both are direct `return idna.Lookup.ToASCII(v)` lines inside an `idnaASCII`
helper. The structural rule sees "no recheck in this block" and alerts. That
is correct for the bug class, but it is also correct for any
trivially-wrapping helper that legitimately just returns the ASCII form for
downstream consumers to validate. The structural rule cannot tell the two
apart.

The taint approach can: it fires only when an *untrusted-input-tainted* value
flows through `ToASCII` to a *network sink*. CodeQL's inter-procedural
taint catches this naturally (the taint enters `idnaASCII` from the caller's
URL.Hostname source, propagates through the wrapper's body, exits as the
wrapper's return value, flows into the caller's `net.JoinHostPort` sink).
OSS Semgrep cannot but is honest about it.

## Recommendation in three layers

### Layer 1: CodeQL primary

Keep the existing `IdnaIpLiteralSmuggle.ql` and `IdnaIpLiteralSmuggle.qll`
shape (stateful taint with `TPreIdna` / `TPostIdna`). Verify by extracting a
CodeQL DB for `golang.org/x/net/http/httpproxy` and confirming the query
fires on `canonicalAddr` (which it should, because the inter-procedural
default propagates the URL.Hostname taint through `idnaASCII` and into
`net.JoinHostPort`). If the verification turns up an unmodelled wrapper
shape, add an `isAdditionalFlowStep` for it.

This is the rule the registry submission should lead with. The CodeQL
community pack already accepts this shape — the closest analog,
`go/cmd-injection`, ships at `@precision high` with explicit wrapper
modelling, and the `BadRedirectCheck.ql` query carries an
`isBadRedirectCheckOrWrapper` predicate to handle one-deep helpers.

### Layer 2: Semgrep OSS taint as direct-call precision sweep

Keep the existing `idna-ip-literal-smuggle.yaml`. It is already correctly
designed as a high-precision direct-call sweep with explicit
`pattern-sources` that match untrusted-input shapes. Document that operators
should expect zero alerts on most codebases because real callers wrap, and
that the rule is a complement to (not a replacement for) the CodeQL companion.

The Pro yaml's `interfile: true` is still load-bearing for operators who
have Pro Engine; do not remove it. The experimental yaml's relaxed
field-name source set is opt-in only; ship it as documented.

### Layer 3: do not ship a blanket structural rule

Resist the temptation to add a `pattern-not-inside`-driven structural rule
that flags every ToASCII without recheck. The empirical 30/31 corpus
result, combined with P3's enumerated legitimate-non-recheck use cases
(PSL walkers, registrar pipelines, TLS-cert managers), makes this a
~95%-FP rule in the strict bug sense.

If the registry pushes back on the OSS taint rule's narrow source set,
the right response is "Pro Engine extends recall through wrappers; the OSS
tier deliberately stays narrow because the alternative is unacceptable
FP density." This is the same trade-off `BadRedirectCheck.ql` documents
and accepts.
